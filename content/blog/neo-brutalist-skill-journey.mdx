# From Skill to Production: Building and Using My First Agent Skill

Building design systems is one thing. Making them reusable across projects is another. But what if your design system could be an agent skill that automates entire UI refactors? That's exactly what I discovered while building my first skill and using it to redesign this portfolio.

## The Problem: Repetitive Design Refactors

Every new project started the same way:
- Manually re-implement the same token system
- Copy-paste component styles from previous projects
- Rebuild the same typography hierarchy
- Maintain separate design docs that drift from code

I needed something better: a **reusable, design-system-first skill** that agents could understand and apply.

## Building the Skill: neo-brutalist-ui-system

### Design Philosophy

The skill needed to be:
1. **Design-system-first**, not template-first
2. **Layout-preserving** by default (respect existing architecture)
3. **Token-driven** with strong typography hierarchy
4. **Automatable** via clear DNA extraction and implementation flow

### Core Principles

**Neo-brutalist design DNA:**
- Strong borders (`2px`), hard-edged surfaces
- Typography-driven hierarchy (not decorative)
- Hard shadows with deliberate offset
- Direct interactions (translate + shadow reduction)

**Token model:**
```css
--background, --surface, --surface-strong
--foreground, --foreground-muted, --foreground-subtle
--border, --border-subtle
--accent, --accent-hover
--shadow-color, --shadow-accent
```

### Skill Structure

The skill includes:
- `SKILL.md`: Complete design system contract (171 lines)
- Design DNA extraction workflow
- Component primitive patterns (button, card, input, dialog, tabs)
- Layout patterns for landing/marketing/portfolio/dashboard
- Theme behavior (light + dark support)
- Generated token script (`generate_blocky_palette.py`)

### Publishing to skills.sh

**Repository setup:**
```bash
# Created ui-skills repo
mkdir ui-skills
cd ui-skills
git init

# Added the skill
mkdir neo-brutalist-ui-system
```

The skill is now publicly available for anyone to use.

## Lessons Learned

### What Worked

1. **Design system as code**: The skill contract made intent explicit
2. **DNA extraction**: Understanding existing structure before changing
3. **Layout-preserving default**: Faster implementation, less risk
4. **Token script**: Quick color palette generation with proper contrast

### Challenges

1. **Context preservation**: Agent needed to understand full app structure
2. **Incremental rollout**: Testing changes section-by-section vs. all-at-once

## Code Example: Button Primitive

```typescript
import { cva, type VariantProps } from "class-variance-authority"

const neoButtonVariants = cva(
  "inline-flex items-center justify-center font-bold uppercase tracking-wider transition-all border-2",
  {
    variants: {
      variant: {
        default: "bg-[var(--nb-surface)] text-[var(--nb-foreground)] border-[var(--nb-border)]",
        primary: "bg-[var(--nb-accent)] text-white border-[var(--nb-accent)]",
        outline: "bg-transparent text-[var(--nb-foreground)] border-[var(--nb-border)]",
      },
      size: {
        sm: "px-3 py-1.5 text-xs",
        md: "px-4 py-2 text-sm",
        lg: "px-6 py-3 text-base",
      },
      active: {
        true: "translate-y-[2px] shadow-none",
        false: "shadow-[4px_4px_0px_0px_var(--nb-shadow-color)]",
      },
    },
    defaultVariants: {
      variant: "default",
      size: "md",
      active: false,
    },
  }
)

export const NeoButton = ({ children, variant, size, className, ...props }) => {
  return (
    <button className={cn(neoButtonVariants({ variant, size }), className)} {...props}>
      {children}
    </button>
  )
}
```

This pattern: border-2, hard shadow, uppercase label, translate on active → now consistent across the entire app.
